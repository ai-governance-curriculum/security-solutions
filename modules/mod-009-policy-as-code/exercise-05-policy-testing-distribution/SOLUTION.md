# SOLUTION — Exercise 05: Policy Testing + Distribution Plan

> Read this *after* attempting the learning-side exercise. This file
> is the operational backbone for the policy-as-code program: how
> policies get tested, signed, distributed, audited, and rolled back.
> Most of the design choices are tradeoffs (bundle server vs. GitOps,
> dry-run duration, fail-open vs. fail-closed during outages), and
> this solution makes each call explicitly so a reviewer can argue
> with the choice rather than guess at it.

## 1. Solution overview

The plan covers five operational concerns:

1. **Testing pipeline** — unit, integration, conformance, performance.
2. **Signing + bundling** — per-environment OPA bundles, Cosign-signed.
3. **Distribution** — bundle server (S3 + verification at pull).
4. **Audit-chain integration** — every decision is queryable evidence.
5. **Rollback** — bundle versioning + rehearsed procedure.

The plan assumes the engine choice from Exercise 01 (Gatekeeper), the
policy library from Exercises 02 and 04, and the CI gate from
Exercise 03. None of the structure depends on Gatekeeper specifically;
the bundle server pattern works for any OPA-compatible engine, and
the Kyverno equivalent (PolicyReports + GitOps sync) is sketched in
section 5.

## 2. Worked implementation

### 2.1 Testing pipeline

| Layer | Tool | When run | Pass criterion |
|-------|------|----------|----------------|
| Unit | `opa test` | every PR | 100% of test rules pass; coverage ≥ 80% via `opa test --coverage` |
| Integration | `kind` + Gatekeeper + sample resources | every PR after unit | all expected allow/deny outcomes match; webhook latency p95 < 200 ms |
| Conformance | bundle applied to a shadow-mode cluster fed production-like resources | every release candidate, plus 30-day shadow on new policies | < 5% unexpected denies on representative sample; manual triage of every novel deny |
| Performance | k6 load test against the admission webhook | every release candidate; nightly on a fixed bundle | webhook p95 < 200 ms at 500 req/s; CPU < 1.5 cores; memory stable < 1 GiB |

The acceptable-latency bound (`< 200 ms p95`) and load level (`500
req/s`) are placeholder targets — the right number is whatever leaves
sufficient headroom over current admission throughput.
<!-- follow-up: replace 200ms/500rps with the SmartRecs measured

**Unit tests.**
- Location: `policies/<package>/*.rego` with `test_*` rules.
- Coverage: tracked via `opa test --coverage`, surfaced as a CI
  artifact and Slack-posted on regressions.
- Required: every new rule lands with ≥ 1 happy-path test and ≥ 1
  failure-mode test.

**Integration tests.**
- Cluster: ephemeral `kind` cluster spun up per CI run; Gatekeeper
  installed via Helm with the candidate bundle.
- Test resources: a curated `tests/resources/` tree with one manifest
  per expected outcome (`allow-*.yaml`, `deny-*.yaml`).
- Driver: a small Python harness in `tests/integration/run.py` that
  `kubectl apply --dry-run=server`s each fixture and asserts allow/
  deny matches the filename prefix.
- Pass criterion: zero unexpected outcomes.

**Conformance tests.**
- Source: weekly export of representative resources from prod
  clusters (`kubectl get -A` of Pods, NetworkPolicies, ModelDeployments,
  TrainingJobs), redacted via a fixed pipeline so PII never leaves
  prod.
- Cadence: every release candidate runs the candidate bundle against
  the latest export; any *new* deny that did not occur with the prior
  bundle is flagged for manual triage.
- Tuning loop: new policies always ship in `warn` mode first
  (see §"Warn-then-enforce ramp"), become `deny` after 30 days of
  clean conformance plus owner sign-off.

**Performance tests.**
- Tool: [k6](https://k6.io/) firing AdmissionReview-shaped payloads at
  the Gatekeeper webhook through `kubectl apply`-equivalent flows.
- Acceptable p95: as above; tracked over time so policy growth that
  degrades latency is caught.

**Warn-then-enforce ramp.** A new policy ships with two rule heads:
`warn` (always evaluated, logged) and `deny` (gated behind a feature
flag in `input.data.enforce[<package>]`). The flag flips after 30
days of clean conformance plus owner sign-off:

```rego
deny contains msg if {
    enforced
    # ... violation conditions ...
}
warn contains msg if {
    not enforced
    # ... same violation conditions ...
}
enforced if input.data.enforce[opa.runtime().env.POLICY_PACKAGE]
```

### 2.2 Signing + bundling

**Bundle structure.**

```
bundle/
├── .manifest                     # OPA bundle manifest (auto-generated by `opa build`)
├── policies/
│   └── ... Rego files ...
├── data/
│   ├── allowed_signers.json
│   ├── thresholds.json
│   ├── enforce.json              # per-package enforce flag (warn/deny ramp)
│   └── ...
└── VERSION                       # `<gitsha>-<utc-timestamp>`
```

**Per-environment configuration.**

| Env | Source branch | Signing identity (Cosign keyless OIDC) | Refresh interval |
|-----|---------------|----------------------------------------|------------------|
| dev | `main` | `https://github.com/smartrecs-org/policies/.github/workflows/build.yml@refs/heads/main` | 60 s |
| staging | `staging` (after `main` merge) | same identity, `@refs/heads/staging` | 5 min |
| prod | tagged release `v*` | same identity, `@refs/tags/v*` | 15 min |

The signing identity is bound to the GitHub Actions workflow via
Sigstore's Fulcio-issued certificate (no private key managed). The
verifier (the OPA bundle plugin or the bundle-fetch sidecar) checks
that the signature was issued by the expected workflow and rejects
otherwise.

**Bundle CI workflow** (`.github/workflows/bundle.yml`):

```yaml
name: bundle
on:
  push:
    branches: [main, staging]
    tags: ['v*']

permissions:
  id-token: write    # Sigstore keyless signing
  contents: read

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: open-policy-agent/setup-opa@v2
        with: {version: latest}
      - name: Test policies
        run: opa test policies/ --coverage
      - name: Build bundle
        run: |
          ENV=$(echo "${GITHUB_REF_NAME}" | sed 's|refs/heads/||;s|refs/tags/.*|prod|')
          echo "${GITHUB_SHA}-$(date -u +%Y%m%dT%H%M%SZ)" > bundle/VERSION
          opa build -b bundle/ -o smartrecs-${ENV}.tar.gz
      - name: Sign bundle with Cosign keyless
        uses: sigstore/cosign-installer@v3
      - run: |
          cosign sign-blob --yes \
            --output-signature smartrecs-${ENV}.tar.gz.sig \
            --output-certificate smartrecs-${ENV}.tar.gz.pem \
            smartrecs-${ENV}.tar.gz
      - name: Upload to bundle server (S3)
        run: |
          aws s3 cp smartrecs-${ENV}.tar.gz \
            s3://smartrecs-policy-bundles/${ENV}/$(cat bundle/VERSION)/bundle.tar.gz
          aws s3 cp smartrecs-${ENV}.tar.gz.sig \
            s3://smartrecs-policy-bundles/${ENV}/$(cat bundle/VERSION)/bundle.tar.gz.sig
          aws s3 cp smartrecs-${ENV}.tar.gz.pem \
            s3://smartrecs-policy-bundles/${ENV}/$(cat bundle/VERSION)/bundle.tar.gz.pem
          # The "current" pointer is the engine's lookup path:
          echo $(cat bundle/VERSION) | \
            aws s3 cp - s3://smartrecs-policy-bundles/${ENV}/current
```

### 2.3 Distribution

**Decision: bundle server (S3 + signature verification at pull),
not GitOps push.** Reasoning:

- OPA / Gatekeeper natively poll a bundle endpoint; the bundle plugin
  handles ETag / version skipping, retry, and exponential backoff.
- GitOps push couples policy rollout to cluster-state reconciliation
  cadence and adds a per-cluster ArgoCD app to manage. For policy,
  the engines themselves are already the right "pull" actor.
- S3 + signed URLs is a cheap, well-understood storage; replication
  to a secondary region is one S3 setting.

**OPA pull configuration** (`/etc/gatekeeper/bundle.yaml` or the
equivalent Helm value):

```yaml
services:
  - name: smartrecs-bundles
    url: https://smartrecs-policy-bundles.s3.amazonaws.com
    credentials:
      s3_signing:
        region: us-east-1
        service: s3
        environment_credentials: {}

bundles:
  smartrecs:
    service: smartrecs-bundles
    resource: prod/current/bundle.tar.gz
    polling:
      min_delay_seconds: 60
      max_delay_seconds: 120
    signing:
      keyid: cosign-keyless
      # Engine-side cosign verifier checks the .sig + .pem next to
      # the bundle artifact; certificate identity must match the
      # release workflow OIDC subject (see Exercise 03 / Module 03).
```

**Failure modes.**

| Failure | Engine behavior | Operational response |
|---------|-----------------|----------------------|
| Bundle server (S3) down | OPA keeps serving the *last successfully verified bundle*; logs `bundle_request_failed`. Engine does NOT fail open at admission. | Page on `bundle_request_failed > 0 for 5m`. Verify S3 status; if extended, rehearsed rollback to a previous bundle from secondary region. |
| Stale bundle (> 1 hour without refresh) | OPA continues to enforce the cached bundle. New policies do not deploy. | Metric `bundle_age_seconds` SLO-monitored; warn at 1 h, page at 4 h. |
| Signature verification fails | OPA refuses to load the bundle. Cached bundle remains in force. | Page; this means a bundle was tampered or the CI signing identity changed unexpectedly. |
| Engine pod itself down | Admission controller failure policy decides: `failurePolicy: Fail` for SmartRecs (fail-closed at admission). Existing pods unaffected. | Multiple replicas of the admission controller; PodDisruptionBudget; restart on the orchestrator. |

The decision to set `failurePolicy: Fail` on the admission webhook is
deliberate: a brief admission outage is preferable to ungoverned
pods entering the cluster. The cost is that a long outage stops all
deploys until the engine recovers; this is bounded by the engine HA
plus the ability to bypass via cluster-admin in a documented
break-glass procedure.

### 2.4 Audit-chain integration

**Entry schema** (one row per policy decision):

```json
{
  "version": "1",
  "timestamp": "2026-05-29T13:24:11.847Z",
  "workload_identity": {
    "subject_uri": "spiffe://smartrecs.example/ns/prod/sa/recs-api",
    "namespace": "prod-east",
    "service_account": "recs-api"
  },
  "resource": {
    "kind": "Pod",
    "name": "recs-api-abcd1",
    "namespace": "prod-east",
    "uid": "..."
  },
  "policy": {
    "package": "security.images.signed",
    "bundle_version": "abcdef0-20260529T120000Z",
    "rule_severity": "deny"
  },
  "decision": "deny",
  "violations": [
    "container image ghcr.io/x/y:rogue is not signed by an approved SmartRecs signer"
  ],
  "engine": {"name": "gatekeeper", "version": "3.16.0"},
  "signing": {
    "identity": "audit-chain-signer@smartrecs.example",
    "signature": "MEUCIQD..."
  }
}
```

**Pipeline.** Gatekeeper's audit logs and admission decision events
are shipped via Fluent Bit to a Kafka topic, transformed into the
schema above by a small consumer, and signed before persistence into
an append-only store (S3 + Object Lock for SOC 2 evidence; a hot
queryable copy in ClickHouse for ad-hoc audit queries).

**Query patterns.**

1. *All deny decisions in the last 7 days for namespace prod-east:*

```sql
SELECT timestamp, resource.name, policy.package, violations
FROM audit_chain
WHERE decision = 'deny'
  AND resource.namespace = 'prod-east'
  AND timestamp > now() - INTERVAL 7 DAY
ORDER BY timestamp DESC;
```

2. *All policy changes deployed in the last 30 days:*

```sql
SELECT DISTINCT policy.package, policy.bundle_version, MIN(timestamp) AS first_seen
FROM audit_chain
WHERE timestamp > now() - INTERVAL 30 DAY
GROUP BY policy.package, policy.bundle_version
ORDER BY first_seen DESC;
```

3. *The policy version that decided a specific rejection (incident
   forensics):*

```sql
SELECT policy.bundle_version, violations, signing.signature
FROM audit_chain
WHERE resource.uid = '<the UID from the incident>'
  AND decision = 'deny';
```

### 2.5 Rollback

**Procedure.**

1. Identify the bad bundle version from the audit chain (the
   `policy.bundle_version` on the offending denies).
2. Identify the prior known-good version
   (`aws s3 ls s3://smartrecs-policy-bundles/<env>/ --recursive`).
3. Update the `current` pointer:

   ```bash
   echo "<prior-good-version>" | \
     aws s3 cp - s3://smartrecs-policy-bundles/<env>/current
   ```

4. Wait for the bundle poll interval (≤ 5 min in prod) for engines
   to pick up.
5. Confirm via the audit chain that new decisions are using the
   rolled-back `bundle_version`.
6. Open a post-incident issue in the policies repo referencing the
   bad version; the offending policy ships back into `warn` mode for
   re-tuning.

**Authority.** The rollback `aws s3 cp` is allowed for members of
the `policy-oncall` IAM group (security lead + 2 SREs). The
audit-chain ingester logs the rollback event with the IAM identity
that pushed the new `current` pointer.

**Communication.** A rollback fires an SRE incident channel post
automatically (the S3 event triggers a Lambda that posts to Slack +
PagerDuty). The policy author is paged regardless of business hours.

**Rehearsal.** Quarterly. Acceptance criteria for rehearsal:

- A staging bundle is intentionally broken (one policy with an
  always-deny rule).
- Time from rollback decision to engine seeing the rolled-back
  bundle ≤ 10 minutes.
- Audit chain shows the bad bundle's decisions and the rollback
  event without manual reconstruction.
- Two SREs (not the security lead) successfully execute rollback
  unaided.

### 2.6 Kyverno equivalent (short form, for engine-portability)

If SmartRecs revisits Exercise 01 and switches to Kyverno, this plan
maps as:

- **Bundles → GitOps.** Kyverno does not have an OPA-style bundle
  plugin; ArgoCD syncs `ClusterPolicy` objects from a signed Git
  commit. Sigstore signing on commits (or signed image-pull from an
  OCI registry storing the policy bundle) covers the integrity
  story.
- **Audit-chain → PolicyReport.** Kyverno produces `PolicyReport`
  CRDs natively; ship those into the same audit-chain pipeline via a
  controller that watches `PolicyReport` events.
- **Rollback → Git revert + ArgoCD sync.** Procedure is the same
  structurally; the authority and rehearsal sections do not change.


## 4. Rubric / review checklist

| Area | Weight | What "pass" looks like |
|------|--------|-----------------------|
| Testing pipeline concrete | 20 | All four layers (unit, integration, conformance, perf) named tools + pass criteria |
| Signing connects to Module 03 / Cosign | 15 | Keyless OIDC identity bound to a specific workflow; verified at pull |
| Bundle structure documented | 10 | policies/ + data/ + manifest + VERSION; per-environment table |
| Distribution choice justified | 10 | Bundle server vs. GitOps decision named with reasoning |
| Failure modes complete | 10 | Server-down, stale, signature-fail, engine-down all handled |
| Audit-chain schema queryable | 15 | Schema includes workload identity, policy version, decision, signature; 3 example queries |
| Rollback rehearsable | 10 | Procedure, authority, communication, rehearsal cadence + acceptance criteria |
| Warn-then-enforce ramp | 5 | New policies start in warn for ≥ 30 days, flip via data flag |
| Risks + mitigations named | 5 | Stale-policy / perf regression / operator error addressed |

Pass threshold: **75 / 100**.

Auto-fail conditions:

- "Test the policies somehow."
- No signing on bundles.
- Audit-chain entries vague ("log the decision") or not queryable.
- No rollback procedure or no rehearsal.
- Engine fails open silently in any of the listed failure modes
  without an explicit justification.

## 5. Common mistakes

1. **Conflating coverage with quality.** `opa test --coverage`
   reports rule coverage, not behavioral coverage. A 100% coverage
   library can still miss edge cases; conformance tests are what
   catch behavioral gaps.
2. **Skipping the conformance step in shadow mode.** A new policy
   that looked clean in unit tests fires unexpected denies the
   minute it sees real production resources. Shadow mode (warn-only
   evaluation against the existing fleet for 30 days) is what makes
   the enforcement ramp safe.
3. **Letting the bundle server be the single point of failure.**
   S3 has its own SLOs, but a misconfigured IAM policy or a
   regional outage can still block bundle refresh. The cached
   bundle behavior in OPA is the mitigation; ensure
   `persist: true` so cached bundles survive engine restarts.
4. **Fail-open at admission without a written justification.**
   `failurePolicy: Ignore` looks safer at first glance but
   silently bypasses every policy when the engine is briefly
   unavailable. Default to `Fail` and document the case where you
   would deviate.
5. **Bundles with no version.** Without `bundle_version` in the
   audit chain, you cannot answer "which policy decided this
   rejection?" — and the rollback target becomes guesswork.
6. **Signing the bundle but not verifying at pull.** The CI signs;
   if the engine accepts unsigned bundles, the signature is
   evidence only, not enforcement. Verify at pull and refuse on
   failure.
7. **Rollback authority concentrated in one person.** If only the
   security lead can roll back, every middle-of-the-night incident
   waits for that one pager. Two-person quorum for normal hours,
   one-person for incident response, both documented.
8. **Not rehearsing rollback.** A rollback that has never been
   executed is a rollback that will fail the first time it is
   actually needed.
9. **Conformance metrics that grow forever.** "Every new deny in
   conformance is reviewed" works at 5 denies/day but breaks at 500;
   put a triage SLA on it and tune the policy rather than
   accumulating a backlog.
